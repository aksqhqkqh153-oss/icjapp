이청잘 앱 - 구글 플레이 스토어 앱 등록 절차 및 준비사항 메모

[폴더명 안내]
- 보편적으로 프로젝트 문서/메모는 docs 폴더로 관리합니다.
- 단순 메모는 docs 안에 .txt 또는 .md 파일로 두는 방식이 가장 일반적입니다.

[현재 프로젝트 상태 요약]
- 현재 프로젝트는 FastAPI + React(Vite) 기반 웹앱/SPA 구조입니다.
- Google Play에 직접 업로드 가능한 Android Studio 프로젝트(AAB 생성 구조)는 포함되어 있지 않습니다.
- Play 등록 전에는 Android 패키징(예: Capacitor, Trusted Web Activity, WebView Shell 중 1개)과 공개 배포용 서버가 추가로 필요합니다.

[Google Play 등록 전 필수 준비]
1. 배포 형태 결정
   - 웹앱을 Android 앱으로 패키징할 방식 선택
   - 권장: Capacitor(하이브리드 앱) 또는 TWA(웹 중심)

2. 백엔드 공개 배포
   - 로컬 127.0.0.1 서버로는 심사 불가
   - HTTPS 도메인 + 운영 서버 필요
   - 앱/백엔드 접속 URL 고정 필요

3. 계정/개인정보 정책 정비
   - 로그인/회원가입/프로필/위치/채팅 기능이 있으므로 개인정보처리방침 URL 필요
   - 계정 생성 가능 앱이면 앱 안 계정 삭제 경로 + 웹 계정 삭제 요청 URL 필요

4. 스토어 등록용 자료 준비
   - 앱 이름
   - 짧은 설명/상세 설명
   - 대표 아이콘(512x512)
   - 피처 그래픽(1024x500)
   - 휴대폰 스크린샷
   - 개인정보처리방침 URL
   - 고객지원 이메일/웹사이트(권장)

5. 기술/정책 체크
   - Android App Bundle(.aab) 생성
   - targetSdk 최신 요구사항 충족
   - 실제 사용하는 권한만 선언
   - 위치/파일/알림/카메라/마이크 사용 시 사유와 실제 기능 일치 필요
   - 테스트 계정 준비

[이 프로젝트에서 추가 보완 권장사항]
- 앱 패키징 구조 추가(android/ 폴더 또는 capacitor 설정)
- 앱 아이콘/스플래시/패키지명/버전명/버전코드 추가
- 개인정보처리방침 페이지 작성 및 공개 URL 확보
- 계정 삭제 기능(앱 내부 + 외부 웹 링크) 구현/노출
- 운영용 로그인/회원관리/비밀번호 재설정 흐름 점검
- Play 심사용 데모 계정 문서화
- Data safety 작성용 데이터 수집 항목 표 정리
- 서버/DB/파일 업로드 운영 구조 정리
- 앱 서명 키 관리 계획 수립

[심사 시 자주 걸리는 포인트]
- 앱 설명과 실제 기능 불일치
- 로그인 필요 앱인데 심사 계정 미제공
- 개인정보처리방침 누락 또는 내용 부실
- 계정 생성 가능하지만 삭제 경로 없음
- 권한만 선언하고 실제 기능/사용 이유가 불명확함
- 테스트 중 앱 크래시/빈 화면/서버 미연결
- 로컬 서버/임시 주소 기반 동작

[권장 다음 단계]
1) Android 패키징 방식 확정
2) HTTPS 운영 배포
3) 개인정보처리방침/계정삭제 준비
4) App Bundle 생성
5) Play Console 등록 자료 입력
6) 내부 테스트 -> 필요 시 폐쇄 테스트 -> 심사 제출
